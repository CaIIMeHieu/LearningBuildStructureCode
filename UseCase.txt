-------------------Dispatch Event-------------------
- Thực hiện dispatch event ở pipeline transaction của MediatR ( tối ưu việc sau mỗi handler lại gọi dispatch, chúng ta gọi dispatch ở một chỗ )
- Có 2 options khi gọi dispatch event :
	1. Dispatch trước transaction.Complete() : nếu có lỗi xảy ra trong transaction thì event sẽ không được dispatch
	2. Lưu cả event vào database ( outbox pattern ) và dispatch sau khi transaction.Complete() : nếu có lỗi xảy ra trong transaction thì event vẫn được lưu vào database và sẽ được dispatch sau với worker

- Trade off :
	1. Đơn giản hơn, không phải lưu database, nhưng do việc thực hiện dispatch cùng với lưu db nên sẽ giam connection db cho đến khi transaction hoàn thành
	   nếu có những handler như là Email handler thì sẽ làm chậm transaction ( đã implement )
	2. Phức tạp hơn, phải lưu database, nhưng sẽ không giam connection db cho đến khi transaction hoàn thành
	   , có thể xử lý retry nếu có lỗi xảy ra trong quá trình dispatch event ( chưa implement )


------------------Rotate Refresh Token-----------------------
- Khi một người dùng ở một thiết bị sử dụng refresh token để lấy access token mới, chúng ta sẽ tạo một refresh token mới và vô hiệu hóa refresh token cũ
- Điều này giúp tăng cường bảo mật bằng cách giảm nguy cơ bị lạm dụng refresh token nếu nó bị đánh cắp
Ví dụ : attacker đánh cắp được refresh token của user, nếu attacker sử dụng refresh token đó để lấy refreshtoken thì khi người dùng gọi refreshToken sẽ bị force logout
Attacker dùng abc123 → server revoke abc123, issue xyz789
User A dùng abc123 → fail vì đã bị revoke → biết ngay token bị đánh cắp → force logout


-----------------JWT đảm bảo tính nhất quán của dữ liệu-------------
token được chia làm 3 phần header, payload, signature
- Header : chứa thông tin về thuật toán mã hóa
- Payload : chứa thông tin về user và các claim
- Signature : được tạo ra bằng cách mã hóa header và payload bằng một secret key, giúp đảm bảo tính toàn vẹn của token
Nếu attacker sửa payload thì khi server có secretkey kí lại token sẽ không khớp với signature, server sẽ phát hiện token đã bị sửa và từ chối yêu cầu


-----------------OAuth2 Authorization Code Flow---------------------
Click Login with Google -> Redirect tới trang đăng nhập Google -> Google redirect về redirectURI của application với authorization code 
-> Application gửi authorization code và client secret tới Google để lấy access token và refresh token ( đảm bảo an toàn vì cơ chế mã hóa nội dung truyền tải của https )
-> Application sử dụng access token để tạo/tìm user trong db 
-----------------Setup Google OAuth credential----------------------
Vào console.cloud.google.com
Tạo project mới (hoặc dùng project có sẵn)
APIs & Services → Credentials → Create OAuth 2.0 Client ID
Application type: Web application
Authorized redirect URIs: 7153https://localhost:7xxx/signin-google (port của project bạn)
